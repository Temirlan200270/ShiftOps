# Telegram-бот — команды, диплинки, уведомления

## Идентичность бота

- Один глобальный бот: `@ShiftOpsBot`.
- Поверхность взаимодействия = а) команды, б) запуски Web App,
  в) инлайн-кнопки (одобрить / отклонить waiver).

## Команды

| Команда | Аудитория | Поведение |
| - | - | - |
| `/start` | все | Если не привязан: поприветствовать + «попросите админа добавить вас». Если привязан: открыть TWA. |
| `/start inv_<token>` | диплинк приглашения | Принимает одноразовое приглашение и привязывает `telegram_accounts.user_id`. |
| `/myshift` | оператор | Диплинк в экран текущей смены. |
| `/today` | админ / собственник | Краткая текстовая сводка по сменам сегодняшнего дня по всем локациям. |
| `/help` | все | Короткая справка. Текст подбирается по роли (super-admin / owner / admin / operator). |
| `/team_list` | владелец | Список активных участников организации с подсказками по `/set_role` и `/remove_member`. |
| `/set_role <@username\|tg_id> <admin\|operator\|bartender>` | владелец | Меняет роль участника между `admin`, `operator` и `bartender`. Роль `owner` через эту команду не выдаётся. |
| `/remove_member <@username\|tg_id>` | владелец | Деактивирует участника (soft-delete). |
| `/create_org` | super-admin | FSM создания пустой организации без владельца. |
| `/org_invite <org_uuid> <owner\|admin\|operator\|bartender> [hours]` | super-admin | Создаёт инвайт-ссылку для указанной роли. |
| `/org_delete <название\|uuid>` | super-admin | **Безвозвратно** удаляет организацию и каскадно все связанные данные в БД. Имя — как при `/create_org`, без кавычек; несколько слов подряд. При совпадении имён — только UUID. |
| `/org_set_owner <org_uuid> <tg_user_id>` | super-admin | Назначает / переназначает владельца, существующих owner'ов в этой org понижает до admin. |
| `/org_set_role <org_uuid> <tg_user_id> <admin\|operator\|bartender>` | super-admin | Меняет роль участника в любой организации. |
| `/org_remove_member <org_uuid> <tg_user_id>` | super-admin | Деактивирует участника в указанной организации. |
| `/cancel` | super-admin | Сбрасывает текущий FSM-сценарий. |

`/start`-диплинки используют `t.me/ShiftOpsBot?start=<payload>`, где
`<payload>` это `inv_<token>` для приглашений или `shift_<uuid>` для
«открыть конкретную смену».

### Авторизация для команд управления командой

| Действие | Требуемая роль |
| - | - |
| Просмотр списка | `admin`, `owner`, super-admin (TWA) |
| Смена ролей | `owner`, super-admin |
| Деактивация участника | `owner`, super-admin |

Платформенный super-admin определяется по `super_admin_tg_id` в settings и
никаких записей в БД не требует. Защищён от изменений на уровне организации:
владелец не может ни сменить ему роль, ни удалить.

### TWA vs бот (управление командой)

- **Основной канал:** TWA (экран «Команда»), `POST /api/v1/team/members/{user_id}/role`.
- **Бот — fallback:** `/team_list`, `/set_role`, `/remove_member` остаются для
  плохой сети или привычки владельца; не выкидываем из slash-меню без причины.

## Профиль бота (BotFather)

При старте API (`shiftops_api.main.lifespan`) выполняется идемпотентный вызов
`setMyDescription`, `setMyShortDescription` и `setMyCommands` — тексты лежат в
[`infra/telegram/bot_profile.py`](../apps/api/shiftops_api/infra/telegram/bot_profile.py).
Описание показывается в пустом чате до нажатия Start, короткое — в карточке
бота, а команды попадают в slash-меню. Ошибка обращения к Telegram логируется
и не блокирует запуск API.

## Inline callbacks

- Кнопки одобрить / отклонить waiver, отправляемые в личку админа:
  `callback_data = "waiver:<task_instance_id>:approve"` /
  `callback_data = "waiver:<task_instance_id>:reject"`.

## Матрица уведомлений

### Оператор (личка)

- **T-30 мин до запланированного начала:**
  «⏰ Твоя смена в %location% начинается в %HH:MM%. Нажми, чтобы открыть.»
- **T+15 мин после запланированного начала, без actual_start:**
  «⚠️ Смена должна была начаться 15 минут назад. Нажми, чтобы стартовать.»
- **T-60 мин до запланированного конца:**
  «🔔 До закрытия 1 час. Осталось %N% критических задач.»
- **Waiver одобрен / отклонён:**
  «✅ Освобождение по %title% одобрено.» / «❌ Отклонено. Выполни задачу.»

### Админ (групповой чат)

- **Смена открыта:** «🟢 [%location%] %name% начал %template_name% в %HH:MM%.»
- **Пропущена критическая задача:** «🚨 [%location%] КРИТ пропущена:
  %title% — %name%.»
- **Подозрительное фото:** «⚠️ [%location%] phash-коллизия по %title% —
  %name%. <Открыть отчёт>» (с диплинк-кнопкой).
- **Запрошен waiver:** «🛑 [%location%] %name% просит освобождение:
  %title% — %reason%.» с кнопками Approve / Reject.
- **Смена закрыта (чисто):** «✅ [%location%] %name% закрыл %template_name%:
  18/20, без нарушений.»
- **Смена закрыта (с нарушениями):** «🟠 [%location%] %name% закрыл с
  нарушениями: пропущено %list%.»
- **Media-группа в конце смены:** до 10 фотографий в одном альбоме с
  подписью-сводкой.
- **Вакантный слот в зоне риска (Эфир / `vacant_at_risk`):** не только
  пассивный список в TWA, но и **проактивное** сообщение в группу
  админов, если запланированная смена без `operator_user_id` и до
  `scheduled_end` остаётся меньше горизонта (по умолчанию порядка 120 мин,
  конфиг `live_vacant_horizon_min`). Пример смысла: «Слот [шаблон] /
  [пост] в [локации] заканчивается в HH:MM, оператор не назначен».
  **Орг. SLA (пилот):** реакция менеджера в течение ~15 минут с момента
  сообщения (назначение, claim, перенос) — см. `PRD.md` §5.5.

### Собственник (личка)

- Зеркало группы админов.
- **Ежедневный дайджест 09:00 (по TZ локации):**
  ```
  📊 ShiftOps daily digest — 27 апр
  • Bar Aurora: 2 смены, 1 нарушение, балл 87%
  • Bar Polaris: 3 смены, 0 нарушений, балл 96%
  ```

## Стратегия rate-limit

- Все исходящие сообщения кладутся в TaskIQ-задачу
  `send_telegram_message`.
- Token bucket per-chat: максимум 1 сообщение/сек на чат. (Telegram
  разрешает 30/сек на бота, но на отдельный чат лимит жёстче — 1/сек
  безопасный потолок.)
- Глобальный bucket: 30 сообщений/сек на бота в сумме.
- Ретрай 429: экспоненциальный backoff с джиттером, максимум 5 попыток.
  Дальше — лог в Sentry и продолжаем.

## Стратегия пакетирования

- Некритические отчёты по задачам не шлются по одному; они
  накапливаются и выходят как `sendMediaGroup` (≤ 10 фото) на закрытии
  смены.
- Пропуски критических задач и подозрительные фото уходят мгновенно —
  это сигнал пользователю, который нельзя задерживать.

## Настройка webhook

```bash
curl -X POST "https://api.telegram.org/bot${TG_BOT_TOKEN}/setWebhook" \
  -d "url=${API_PUBLIC_URL}${TG_WEBHOOK_PATH}" \
  -d "secret_token=${TG_WEBHOOK_SECRET}"
```

Хендлер webhook'а проверяет `X-Telegram-Bot-Api-Secret-Token` против
`TG_WEBHOOK_SECRET` до того, как передать апдейт в `Dispatcher` aiogram.
