# Мастер-промпт: интерактивное HTML-демо ShiftOps (один файл)

Этот документ — **готовый запрос для нейросети** (ChatGPT, Claude, и т.д.) и краткая инструкция, **какие файлы репозитория прикладывать**. Сгенерированный результат: один автономный `index.html`, открывается в браузере без сборки.

---

## Как пользоваться

1. Скопируй блок **«Мастер-промпт (вставить в чат)»** ниже.
2. В интерфейсе модели **прикрепи файлы** из чеклиста (drag-and-drop или @-упоминания).
3. При необходимости дополни список прототипов своими `прототипы/ai_studio_code (N).html`.
4. Дождись одного цельного HTML-файла; сохрани как `shiftops-demo.html` и открой локально.

## Мастер-промпт (вставить в чат)

Скопируй всё от «РОЛЬ» до конца раздела «КРИТЕРИИ ПРИЁМКИ».

```
РОЛЬ И ЦЕЛЬ

Ты — ведущий fullstack-разработчик и UI/UX-дизайнер с опытом премиальных интерактивных прототипов. Твоя задача — выдать ОДИН самодостаточный HTML-файл (включая встроенный JS и при необходимости <style>), который работает как интерактивный демо-движок продукта ShiftOps («Linear для HoReCa» — оперативный контроль смен в HoReCa, Telegram Web App в проде).

Файл должен открываться двойным кликом в браузере, без сборки и без локального сервера.

ИЕРАРХИЯ ИСТОЧНИКОВ ИСТИНЫ (строго)

1) Документ пользователя «ShiftOps: Дизайн-система и Спецификация UI (v1.2 PRO)» — высший приоритет для визуальных токенов там, где они заданы явно (в частности Base Background и radial glow).
2) Прикреплённые пользователем файлы из репозитория (README.md, docs/*.md, прототипы HTML) — канон продуктовой логики, сценариев и терминов.
3) Общие знания модели — ТОЛЬКО если в прикреплённых файлах нет ответа; не противоречь пунктам 1–2.

КОНФЛИКТЫ МЕЖДУ ДОКАМИ

Если два прикреплённых источника расходятся, не выбирай молча. В самом верху HTML-файла добавь комментарий <!-- SOURCE_RESOLUTION: ... --> с таблицей: «поле / выбранное значение / источник-победитель / краткое обоснование». Пример осмысленного конфликта: DESIGN_SYSTEM.md задаёт bg-base #0B1220, а v1.2 PRO — #000000: используй правило из раздела «Визуальный гибрид» ниже и зафиксируй это в SOURCE_RESOLUTION.

ВИЗУАЛЬНЫЙ ГИБРИД (зафиксировано для этого демо)

- Фон сцены: #000000 (OLED), плюс мягкий radial glow как в v1.2 PRO (например оттенки в духе #1E293B → #0B1220), чтобы сцена не была плоской.
- Glass surface карточек: rgba(22, 33, 58, 0.6), backdrop-filter blur 24px.
- Hairline border: 1px solid rgba(255, 255, 255, 0.08). Если в DESIGN_SYSTEM указано 0.06 — для демо используй 0.08 и упомяни расхождение в SOURCE_RESOLUTION.
- Primary #4F7CFF; Success #30A46C; Warning #F5A524; Critical #E5484D — согласованы с docs/DESIGN_SYSTEM.md и v1.2.
- Шрифт: Inter (Google Fonts). Заголовки 700–800, letter-spacing ~ -0.04em; body 400–500, ~ -0.01em. Опционально JetBrains Mono для таймкодов/«хешей» в модалке камеры.

ПРОТОТИПЫ HTML

Файлы в папке «прототипы» (например shiftops-prototype.html, ai_studio_code*.html) используй как РЕФЕРЕНС КОМПОЗИЦИИ: рамка телефона, сегмент-переключатели, группировка экранов, плотность UI. НЕ копируй устаревшие hex-цвета из старых прототипов — перерисуй под токены v1.2 + гибрид выше.

ТЕХНОЛОГИЧЕСКИЙ СТЕК (обязательно)

- Один HTML-файл.
- Tailwind CSS через официальный CDN (play CDN).
- Иконки: Lucide через CDN (например unpkg/esm с importmap или совместимый способ без сборщика).
- Анимация успеха: canvas-confetti через CDN; вызывай при выполнении условия ниже.
- Логика и состояние: чистый JavaScript без фреймворков.
- Запрещено: npm-сборка, React/Vue, отдельные бандлы, которые пользователь должен собирать сам.

TELEGRAM WEBAPP MOCK

Реализуй безопасные заглушки: если window.Telegram?.WebApp отсутствует, подставь объект с методами HapticFeedback.impactOccurred / notificationOccurred (в браузере — лёгкая визуальная отдача или navigator.vibrate, если доступно), setHeaderColor(hex) — меняй CSS-переменную цвета «шапки» демо. В TWA эти вызовы соответствуют манифесту (тёмно-красный при критическом пропуске, тёмно-зелёный при успешном закрытии, базовый тёмный в обычном режиме).

ОБЪЁМ ДЕМО (согласованный)

- Полная app shell: первый кадр — скелетон логотипа + короткая симуляция «handshake» (400–900 мс), без блокировки всего UI после перехода к демо.
- Переключатель роли: Operator / Owner (аккуратный, может быть в «настройках демо» или компактный тумблер).
- Нижняя навигация высотой ~80px с учётом safe-area-inset-bottom.
  - Operator: вкладки [Смена], [История], [Профиль] — иконки Lucide; контент S1 реализован на вкладке «Смена»; остальные вкладки могут быть лёгкими заглушками с 2–3 строками текста «Демо: основной фокус на Смене».
  - Owner: вкладки [Обзор], [Команда], [KPI], [Настройки]; основной контент S2 размести на «KPI» или «Обзор» (один раз выбери и последовательно используй); прочие вкладки — заглушки shell-уровня, чтобы не раздувать файл.

ЭКРАН S1 — DASHBOARD ОПЕРАТОРА

- Список задач с секциями Critical, Required, Optional. У critical — левая полоса 4px цвета critical.
- Claim model: до «занятия поста» покажи карточки доступных постов (названия из контекста HoReCa) и кнопку «⚡ Занять пост»; после нажатия перейди в активную смену с задачами.
- Задачи возьми по смыслу из docs/PRD.md и/или docs/PILOT_SMOKE.md (открытие смены, проверка кассы и т.д.) — не выдумывай с нуля, если в файлах есть формулировки.
- Score Engine: каноническая формула V1 из docs/SCORE_FORMULA.md:

  score = 100 * (0.50 * completion + 0.25 * critical_compliance + 0.15 * timeliness + 0.10 * photo_quality)

  Где:
  - completion = done_count / total_count; done включает done и waived (одобренные waiver не штрафуют); skipped и waiver_rejected — не выполнены.
  - critical_compliance = 1.0 только если все critical выполнены или waived; иначе 0.0.
  - timeliness: для демо задай scheduled_end и actual_end симулированно; если опоздание — линейное снижение до 0 за окно 2 часа (как в SCORE_FORMULA.md).
  - photo_quality = photos_unique / photos_total по задачам с требованием фото; подозрительные фото считаются в знаменателе и не в числителе.

  В шапке состояния показывай score с пересчётом при каждом изменении задач. Округляй для отображения разумно (например 1 знак), но confetti срабатывает если отображаемый score > 99.9 (учитывай float).

- Закрытие смены: кнопка активна только если все critical закрыты (done или waived). Если есть незакрытый critical — кнопка disabled и пояснение в тексте (как в PILOT_SMOKE: нельзя закрыть с pending critical).

- Anti-fake (симуляция): для задач с фото по тапу открывай модальное окно «видоискатель», анимацию сканирования ~1.5 с, затем случайный исход Unique или Suspicious. Suspicious помечает вложение как подозрительное для photo_quality. Не отклоняй задачу автоматически — только флаг (как EC-5 в UX_FLOW).

- Waiver flow: кнопка «Не могу выполнить» → чипы причин (например «Сломано», «Нет товара») + комментарий → состояние waiver_pending (янтарь). В демо можно кнопку «Симулировать одобрение админа» спрятать в debug-блоке или долгим нажатием на карточку, чтобы не усложнять UX; если не реализуешь одобрение — опиши в комментарии ограничение демо.

ОФЛАЙН (облегчённая симуляция)

- Не используй IndexedDB и idb-keyval.
- Для «очереди фото» используй sessionStorage/localStorage: помечай задачу бейджем «В очереди», при событии window «online» снимай бейдж (псевдо-синк). Сохраняй минимальное состояние демо, чтобы F5 не обнулял всё без надобности (по желанию — ключ shiftops-demo-state).
- В комментарии укажи связь с EC-1 в docs/UX_FLOW.md.

ЭКРАН S2 — ANALYTICS ВЛАДЕЛЬЦА

- Сетка KPI 2×2 (метрики демо-данных: средний балл, нарушения за 24ч, опционально выручка-заглушка, % чистоты — если нет в доках, пометь как «demo placeholder»).
- Heatmap нарушений: 7 колонок (дни недели), интенсивность по количеству нарушений.
- Leaderboard сотрудников по score; худшие визуально выделены (critical-акцент).

КАЧЕСТВО КОДА

- В ключевых местах JS комментарии вида: // SCORE_FORMULA: … // UX_FLOW: … // PILOT_SMOKE: …
- Доступность: focus-visible, aria-modal на модалках, aria-label на иконках-кнопках.
- Уважай prefers-reduced-motion: отключай или сильно сокращай анимации.

КРИТЕРИИ ПРИЁМКИ (must pass)

[ ] Один .html, без внешних шагов сборки; CDN только указанные классы библиотек.
[ ] Открывается локально; нет ошибок в консоли, мешающих сценарию.
[ ] Роль Operator/Owner переключается; меняются набор вкладок и header (Operator: дата + локация; Owner: аватар/инициалы + роль).
[ ] Скелетон + симуляция handshake 400–900 мс, затем рабочий UI.
[ ] S1: секции Critical/Required/Optional; claim → активная смена; score пересчитывается по формуле V1; закрытие заблокировано при незакрытых critical.
[ ] S1: модалка камеры + скан + Unique/Suspicious; suspicious влияет на photo_quality.
[ ] S1: waiver переводит в янтарный pending; completion/critical_compliance учитывают waived только после «одобрения», если одобрение реализовано — иначе задокументировано в комментарии.
[ ] S2: KPI 2×2, heatmap 7 колонок, leaderboard с выделением худших.
[ ] Confetti при score > 99.9 при выполнении условий демо.
[ ] Офлайн light_sim: бейдж очереди + localStorage/sessionStorage; без IndexedDB.
[ ] В начале файла SOURCE_RESOLUTION при любом осознанном расхождении доков; указать SCORE_FORMULA_VERSION = 1 как в docs/SCORE_FORMULA.md.
[ ] Telegram WebApp mock для haptics и setHeaderColor не ломает обычный браузер.

Выведи ПОЛНЫЙ исходный код HTML одним блоком кода, готовым к сохранению в файл. Не сокращай разумные части «…» — файл должен быть цельным.
```
