# Telegram-бот — команды, диплинки, уведомления

## Идентичность бота

- Один глобальный бот: `@ShiftOpsBot`.
- Поверхность взаимодействия = а) команды, б) запуски Web App,
  в) инлайн-кнопки (одобрить / отклонить waiver).

## Команды

| Команда | Аудитория | Поведение |
| - | - | - |
| `/start` | все | Если не привязан: поприветствовать + «попросите админа добавить вас». Если привязан: открыть TWA. |
| `/start org_<uuid>` | диплинк приглашения | Привязывает `telegram_accounts.user_id` к уже существующему месту. |
| `/myshift` | оператор | Диплинк в экран текущей смены. |
| `/today` | админ / собственник | Краткая текстовая сводка по сменам сегодняшнего дня по всем локациям. |
| `/help` | все | Короткая справка. |

`/start`-диплинки используют `t.me/ShiftOpsBot?start=<payload>`, где
`<payload>` это `org_<uuid>` для приглашений или `shift_<uuid>` для
«открыть конкретную смену».

## Inline callbacks

- Кнопки одобрить / отклонить waiver, отправляемые в личку админа:
  `callback_data = "waiver:<task_instance_id>:approve"` /
  `callback_data = "waiver:<task_instance_id>:reject"`.

## Матрица уведомлений

### Оператор (личка)

- **T-30 мин до запланированного начала:**
  «⏰ Твоя смена в %location% начинается в %HH:MM%. Нажми, чтобы открыть.»
- **T+15 мин после запланированного начала, без actual_start:**
  «⚠️ Смена должна была начаться 15 минут назад. Нажми, чтобы стартовать.»
- **T-60 мин до запланированного конца:**
  «🔔 До закрытия 1 час. Осталось %N% критических задач.»
- **Waiver одобрен / отклонён:**
  «✅ Освобождение по %title% одобрено.» / «❌ Отклонено. Выполни задачу.»

### Админ (групповой чат)

- **Смена открыта:** «🟢 [%location%] %name% начал %template_name% в %HH:MM%.»
- **Пропущена критическая задача:** «🚨 [%location%] КРИТ пропущена:
  %title% — %name%.»
- **Подозрительное фото:** «⚠️ [%location%] phash-коллизия по %title% —
  %name%. <Открыть отчёт>» (с диплинк-кнопкой).
- **Запрошен waiver:** «🛑 [%location%] %name% просит освобождение:
  %title% — %reason%.» с кнопками Approve / Reject.
- **Смена закрыта (чисто):** «✅ [%location%] %name% закрыл %template_name%:
  18/20, без нарушений.»
- **Смена закрыта (с нарушениями):** «🟠 [%location%] %name% закрыл с
  нарушениями: пропущено %list%.»
- **Media-группа в конце смены:** до 10 фотографий в одном альбоме с
  подписью-сводкой.

### Собственник (личка)

- Зеркало группы админов.
- **Ежедневный дайджест 09:00 (по TZ локации):**
  ```
  📊 ShiftOps daily digest — 27 апр
  • Bar Aurora: 2 смены, 1 нарушение, балл 87%
  • Bar Polaris: 3 смены, 0 нарушений, балл 96%
  ```

## Стратегия rate-limit

- Все исходящие сообщения кладутся в TaskIQ-задачу
  `send_telegram_message`.
- Token bucket per-chat: максимум 1 сообщение/сек на чат. (Telegram
  разрешает 30/сек на бота, но на отдельный чат лимит жёстче — 1/сек
  безопасный потолок.)
- Глобальный bucket: 30 сообщений/сек на бота в сумме.
- Ретрай 429: экспоненциальный backoff с джиттером, максимум 5 попыток.
  Дальше — лог в Sentry и продолжаем.

## Стратегия пакетирования

- Некритические отчёты по задачам не шлются по одному; они
  накапливаются и выходят как `sendMediaGroup` (≤ 10 фото) на закрытии
  смены.
- Пропуски критических задач и подозрительные фото уходят мгновенно —
  это сигнал пользователю, который нельзя задерживать.

## Настройка webhook

```bash
curl -X POST "https://api.telegram.org/bot${TG_BOT_TOKEN}/setWebhook" \
  -d "url=${API_PUBLIC_URL}${TG_WEBHOOK_PATH}" \
  -d "secret_token=${TG_WEBHOOK_SECRET}"
```

Хендлер webhook'а проверяет `X-Telegram-Bot-Api-Secret-Token` против
`TG_WEBHOOK_SECRET` до того, как передать апдейт в `Dispatcher` aiogram.
