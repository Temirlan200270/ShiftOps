# Требования к продукту — ShiftOps

> Оперативный контроль смен в HoReCa. Не «ещё один чек-лист», а система,
> которой собственник доверяет вопрос «правильно ли отработал бар сегодня».

## 1. Проблема

Собственники баров / ресторанов / кофеен не могут полагаться на персонал в
части соблюдения процедур открытия/закрытия без личного контроля на месте.
Существующие инструменты делятся на два лагеря:

- Универсальные todo-приложения (Trello, Notion) — нет принуждения, нет
  фотодоказательств, нет напоминаний, легко имитировать выполнение.
- Тяжёлые HoReCa-комбайны (iiko Plus, r_keeper) — дорогие, сложные в
  настройке, рассчитаны на бухгалтерию, а не на линейный персонал.

ShiftOps закрывает середину: смена — это конечный автомат с жёсткими
правилами, каждая критическая задача требует фото (anti-fake на сервере),
а собственник/админ получает оповещения в Telegram в реальном времени,
если что-то идёт не так.

## 2. Персоны

### Собственник («Сергей», 38 лет, владелец 2 баров)

- Вечера проводит дома, но хочет понимать, что всё под контролем.
- Не готов ставить ещё одно приложение — уже живёт в Telegram.
- Хочет утренний дайджест в 09:00: «вчера: 5 смен, 1 с нарушениями,
  средний балл 87%».
- Платит за сервис.

### Админ / менеджер («Анна», 29 лет, управляет 1 баром)

- Собирает шаблоны чек-листов («Утренняя смена», «Передача бара»).
- Утром просматривает фотоотчёты.
- Утверждает / отклоняет освобождения, когда оператор не может выполнить
  задачу.

### Оператор / бармен («Иван», 22 года, линейный персонал)

- 12-часовые смены, часто с мокрыми/холодными руками.
- Приходит за 30 мин до открытия — получает напоминание в Telegram.
- Проходит чек-лист со смартфона: тап по задаче, фото из встроенной
  камеры, дальше. Хочет, чтобы поток был **быстрым** и не мешал работе.

## 3. Jobs to be done

- **JTBD собственника:** «Когда меня нет в баре, дай мне уверенность, что
  заведение открылось/закрылось правильно — чтобы я мог спокойно спать».
- **JTBD админа:** «Когда смена закончилась, дай мне доказательства того,
  что произошло, чтобы я мог поймать проблемы заранее».
- **JTBD оператора:** «Когда начинается моя смена, проведи меня по списку
  дел, чтобы я ничего не забыл и смог уйти вовремя».

## 4. Роли и права

| Роль | Чтение | Запись | Telegram-оповещения |
| - | - | - | - |
| Собственник | вся орг | освобождения, шаблоны, пользователи | личные + группа админов, ежедневный дайджест |
| Админ | свои локации | освобождения, шаблоны, расписание | группа админов, личные критические алерты |
| Оператор | только свои смены | выполнения задач, запросы на освобождение | личные напоминания, без алертов о других |

Авторизация проверяется и в API (декоратор `@requires(role, scope)`), и в
БД (Row-Level Security в Postgres по `app.org_id`).

## 5. Ключевые фичи (MVP)

### 5.1 Шаблоны
В V0 поставляются два хардкоженных шаблона:

- **Утренняя смена** (оператор): уборка, инвентаризация, заготовки —
  8–10 задач.
- **Вечерняя / закрывающая смена**: сверка кассы, уборка, опечатывание.

У каждой задачи есть:

- `title`, `description` (lite-разметка),
- `criticality`: `critical` | `required` | `optional`,
- `requires_photo`: bool,
- `requires_comment`: bool (редко — для «зафиксировать поломку»).

UI-редактор шаблонов **вне V0**, переехал в V1.

### 5.2 Конечный автомат смены

Диаграмма — в `UX_FLOW.md`. Правила:

- Смена создаётся из шаблона + записи расписания. Стартовое состояние —
  `scheduled`.
- `Оператор` нажимает **Старт** → статус `active`, `actual_start = now()`.
- Задача переходит: `pending → done | skipped | waived`.
- **Жёсткая блокировка**: нельзя закрыть смену, пока есть `pending`
  `critical`-задачи. Нужен явный waiver-флоу.
- **Мягкое предупреждение**: закрытие с пропущенными `required`-задачами
  пишет `closed_with_violations` и триггерит алерт админу.
- На закрытии вычисляется числовой `score` (см. `SCORE_FORMULA.md`).

### 5.3 Фотоснимок (anti-fake)

- TWA использует `<input type="file" accept="image/*" capture="environment">`,
  чтобы устройство открывало камеру по умолчанию. Best-effort: на iOS
  возможен fallback в галерею.
- Сервер ставит `captured_at_server = now()` (игнорирует часы клиента).
- Сервер считает `phash` (perceptual hash) и сравнивает с последними 50
  вложениями того же `template_task_id` в той же `location_id`. Если
  расстояние Хэмминга ≤ `ANTIFAKE_PHASH_THRESHOLD` (по умолчанию 5),
  вложение помечается `suspicious=true`, в группу админов уходит алерт с
  обеими фотографиями.
- Опциональный GEO-ассист: если TWA дал доступ к геолокации, сервер
  сравнивает с `location.geo` (радиус 200 м). Несовпадение логируется,
  но не блокирует.

### 5.4 Waiver-флоу

Когда критическую задачу действительно нельзя выполнить:

1. Оператор тапает «Не могу выполнить» → выбирает чип-причину +
   свободный текст + фото причины (поломка, отсутствие товара).
2. Задача переходит в `waived_pending`. Админ получает в Telegram
   сообщение с инлайн-кнопками `Approve` / `Reject`. SLA = 5 мин.
3. **Approve** → статус `waived`, жёсткая блокировка снята.
4. **Reject** или таймаут → статус возвращается в `pending`, оператор
   уведомлён.

### 5.5 Telegram-уведомления

Матрица уведомлений (см. `TELEGRAM_BOT.md`):

- Личные оператора:
  - Напоминание T-30 мин до запланированного начала.
  - «Смена должна была начаться 15 мин назад, ты не нажал Старт» —
    повтор в T+30.
  - «Осталось X критических задач, смена закончится через 1 час» в T-60
    до конца смены.
- Группа админов:
  - «[Локация] смена открыта Иваном».
  - «[Локация] пропущена критическая задача: %title% — Иван»
    (мгновенно).
  - «[Локация] подозрительное фото в %title%» (мгновенно).
  - «[Локация] смена закрыта: 18/20, без нарушений» (на закрытии).
  - В конце смены: media-группа со всеми фотографиями.
  - **Риск по слоту (Эфир / vacant at risk):** запланированная смена без
    оператора, окно до `scheduled_end` входит в горизонт риска (см.
    `live_vacant_horizon_min` на API) — **проактивное** сообщение в чат
    админов: локация, шаблон, пост (`station_label` / `slot_index`),
    время окончания слота. Цель — не заменить экран «Эфир», а дать
    сигнал **до** инцидента. **SLA реакции (организационный, не автоматика):**
    в пилоте ориентир для сменного менеджера — просмотреть чат и
    назначить/передать смену или эскалировать в течение **15 минут** с
    момента уведомления; точное окно фиксируется в runbook пилота.
- Личные собственника:
  - Зеркало группы админов.
  - Ежедневный дайджест в 09:00 (часовой пояс локации).

### 5.6 Аудит-лог

- Каждое изменение статуса пишет неизменяемую строку в `audit_events`.
  Append-only обеспечивает триггер Postgres
  (`BEFORE UPDATE OR DELETE RAISE EXCEPTION`).
- Админ / собственник могут смотреть аудит-лог по смене; оператор — нет.
- Удаление по GDPR / 152-ФЗ — отдельной tombstone-процедурой.

### 5.7 Мультиарендность

- Биллинговая единица — `Organization`.
- RLS-политики Postgres завязаны на `current_setting('app.org_id')`.
  Middleware каждого API-запроса ставит GUC после валидации JWT.

### 5.8 i18n и часовые пояса

- `next-intl` с пакетами RU и EN с первого дня.
- Все таймстемпы смен/задач хранятся в UTC; отображение — по
  `location.timezone`.

### 5.9 Наблюдаемость

- Sentry для ошибок (FE + BE).
- JSON-логи через `structlog`.
- Healthchecks: `/healthz` (liveness), `/readyz` (БД + Redis +
  доступность бота).

### 5.10 Аналитика: разрез по постам (station / slot)

- Для HoReCa типичен вопрос «как бар 1 vs бар 2», а не только средний
  score по локации. В API аналитики (`/v1/analytics/overview`) **запланировано**
  расширение: агрегаты по паре `(station_label, slot_index)` (и/или по
  `slot_index` при пустом label) в рамках локации и периода — после
  стабилизации базовых KPI. До этого отчётность по посту покрывается
  фильтрами **истории смен** (`slot_index`, `station_label`) и карточкой
  смены.

### 5.11 Операционный экспорт для владельца (не GDPR)

- Отдельно от **юридического** экспорта по запросу субъекта данных
  (JSON + zip, см. `SECURITY.md`): **ежедневная** (по расписанию) выгрузка
  для собственника — данные за **предыдущий календарный день** (по TZ
  локации или UTC — фиксируется в реализации), формат **CSV** и/или **PDF**:
  закрытые смены, score, признаки нарушений, при необходимости пост и
  `delay_reason`. Назначение: внутренний аудит и сверка с кассой/сменами
  без ожидания «раз в неделю».

## 6. Вне V0

- UI-редактор шаблонов (V1).
- Аналитический дашборд собственника (V1).
  - V1.x — расширенная owner-аналитика: произвольный период `from`/`to`,
    тумблер сравнения с предыдущим окном (`compare`), drill-down по
    сотруднику (профильный sheet → история с фильтрами `user_id`/
    `location_id`/`from`/`to`), карточки разрезов по шаблонам,
    критичности задач, anti-fake (% подозрительных вложений), SLA
    «опоздание старта» и оператор vs бармен. На каждый блок отдаётся
    density-флаг `ok|low|empty` для подсказки «мало данных».
- Realtime через WebSocket (в V0 — только polling).
- Нативные Flutter-приложения (V3).
- AI-оценка фото (V3).
- Биллинг (V2).
- ~~Запросы на обмен сменами / причина задержки при закрытии~~ — реализовано;
  дальнейшая полировка (подбор смены без UUID, лимиты) по фидбэку.
- White-label-бот для арендаторов (V3).

## 7. Метрики успеха пилота V0

- 1 пилотная организация, 1 локация, 3 оператора, 7 дней в проде.
- ≥ 90% запланированных смен открыты в течение 15 мин от плана.
- ≥ 95% критических задач выполнены (`done` или одобренный `waived`).
- 0 межарендных утечек данных (RLS integration-тест в CI зелёный).
- ≤ 1% флагов `suspicious=true`, которые собственник признаёт ложными.
